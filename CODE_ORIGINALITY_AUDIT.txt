  1. EXECUTIVE SUMMARY
================================================================================

  VERDICT: NO COPIED CODE DETECTED

  All modules demonstrate original implementations built on standard WoW APIs
  and ElvUI's documented toolkit. Where similar features exist in other addons,
  TrenchyUI uses fundamentally different architectures, data sources, or
  approaches. No substantial code blocks (5+ sequential lines) were found to
  match any compared addon.

  One minor note: The Talking Head disable feature (qol.lua lines 14-42)
  mirrors ElvUI's own implementation pattern, but this is because both must
  perform the same Blizzard API calls to achieve the effect — there is no
  alternative approach.


================================================================================
  2. TRENCHYUI vs ELVUI
================================================================================

  Grade: A- (Excellent)

  PROPER API USAGE (POSITIVE FINDINGS):

  • CreateBackdrop / SetTemplate
    Files: damagemeter.lua, elvuf.lua, skin modules
    Uses E:SetTemplate('Transparent'), E:CreateBackdrop() throughout.
    Never reimplements backdrop logic.

  • FontTemplate
    File: qol.lua (lines 127, 132, 143, 144)
    All FontStrings use FontTemplate(fontPath, fontSize, fontOutline).
    Handles ElvUI's SHADOWOUTLINE flag correctly.

  • E:ClassColor
    File: core.lua (lines 106-112)
    TUI:GetClassColor() wraps E:ClassColor() correctly.
    Handles CUSTOM_CLASS_COLORS fallback automatically.

  • Media & Library Access
    Consistently uses E.media.blankTex, E.media.normTex, LSM:Fetch().
    No hardcoded texture paths except for addon-specific assets.

  • Hooking Strategy
    Uses hooksecurefunc(NP, ...) and hooksecurefunc(UF, ...) throughout.
    Never destructively overrides ElvUI functions.
    Preserves ElvUI's execution chain.

  AREAS OF NOTE:

  • Talking Head (qol.lua lines 14-42)
    Mirrors ElvUI's HandleTalkingHead() because both must unregister the same
    events and remove from AlertFrame.alertFrameSubSystems. No hook point is
    exposed by ElvUI for this feature. This is a pragmatic necessity, not a
    design choice.

  • Raw SetBackdrop in qol.lua (lines 435-440, 539-544)
    Used for TUI's own minimap button bar frames. These are TUI-created frames
    (not ElvUI frames), so raw backdrop usage is appropriate. Still uses
    E.media helpers for textures.

  • Raw SetFont in BigWigs skin (line 119)
    Documented exception per CLAUDE.md: "BigWigs skin uses raw SetFont because
    BigWigs applies its own shadow settings afterward."

  3. TDM (DAMAGE METER) vs DETAILS!
================================================================================

  Grade: A (Original)

  FUNDAMENTAL INCOMPATIBILITY:

  TrenchyUI's TDM uses Blizzard's native C_DamageMeter API (introduced in
  patch 12.0 / The War Within). Details! uses COMBAT_LOG_EVENT_UNFILTERED
  parsing. These are completely different data sources that make code reuse
  between the two meaningless.

  TDM APIs:                          Details! APIs:
  ─────────────────────────────────   ─────────────────────────────────
  C_DamageMeter.GetCombatSession     CombatLogGetCurrentEventInfo()
  FromID()                           COMBAT_LOG_EVENT_UNFILTERED
  C_DamageMeter.GetAvailable         Custom CLEU parser
  CombatSessions()                   Client-side event aggregation
  DAMAGE_METER_COMBAT_SESSION_
  UPDATED (event)

  SPECIFIC COMPARISONS:

  • Bar Creation (damagemeter.lua lines 284-342)
    Standard WoW CreateFrame("StatusBar") pattern with FontStrings and
    textures. This pattern appears in dozens of addons. Generic, not derived.

  • Window Management (lines 267-282)
    Custom NewWindowState() with embedded/standalone dual-mode system
    integrated with ElvUI's RightChatPanel. Details! has completely different
    window architecture.

  • Session Handling
    TDM: win.sessionId (stored encounter) vs win.sessionType (Current/Overall)
    Details!: Real-time CLEU parsing, no "session" concept in the same way.

  • Drill-Down (lines 981-1131)
    Built around C_DamageMeter's combatSpells array structure.
    Details! aggregates from raw combat log events.

  • Secret Value Handling (lines 159-170)
    Addresses 12.x ConditionalSecret values. Not a Details! concern since
    Details! predates secret values and uses CLEU.

  • Menu System (lines 561-683)
    Uses ElvUI's E:ComplicatedMenu() / E.EasyMenu(). Standard ElvUI toolkit.

  VERDICT: No copied code. Architecturally incompatible data sources.

  4. CDM (COOLDOWN MANAGER) vs Ayije_CDM / BCDM / ArcUI / CDMCentered
================================================================================

  Grade: A (Original)

  ARCHITECTURAL DIFFERENCES:

  TUI CDM:     Single 642-line file, ElvUI module (E:GetModule('TrenchyUI')),
               stores config in ElvUI's AceDB. Container reparenting system
               that moves Blizzard's native CDM icons into custom TUI-controlled
               frame containers with centered grid layout.

  Ayije_CDM:   Standalone addon (50+ files, ~3000+ lines), custom event system,
               own SavedVariables (Ayije_CDMDB), scoped refresh/callback
               pipeline, pixel-perfect layout engine with PixelUtil, multi-row
               essential/utility sizing, EditMode integration, buff bar viewer,
               custom buff tracking, fading system, defensives/racials/trinkets
               modules.

  BCDM:        Ace3-based framework building power bars, secondary bars, cast
               bars with anchoring. Broader feature set including custom spell
               bars, item bars, trinket tracking.

  CDMx:        Hotkey display system + custom bar framework. Scans action bars
               for keybinds. Different focus entirely.

  ArcUI:       Buff/cooldown tracking with customizable bars. More glow
               variations. No public source repository.

  CDMCentered: Small addon for centering CDM display. No public source.

  DETAILED COMPARISON: TUI CDM vs Ayije_CDM (v3.70, source reviewed 2026-03-09)

  Architecture:
  • TUI: Single file, ElvUI module, ElvUI AceDB config
  • Ayije: 50+ file standalone addon, custom event/callback system with
    RegisterEvent/UnregisterEvent wrappers, scoped RefreshCallback system
    with priority ordering, own SavedVariables
  • No structural overlap.

  Glow System:
  • Both use LibCustomGlow-1.0 (standard library, same public API)
  • TUI (lines 49-115): Simple ApplyGlow/StopGlow driven by
    SpellActivationAlert visibility, hides Blizzard alert via SetAlpha(0),
    per-frame hooks on alert:Show, reusable glowColor table
  • Ayije (Core/Glow.lua, 665 lines): ActionButtonSpellAlertManager hooks,
    debounce drainer frames (OnUpdate-based), buff glow host frames with
    cross-parent anchoring, per-frame data via GetFrameData(), glow cache
    with 20+ config keys, glow color array cache with weak-key metatable,
    separate buff pending stop system with grace period
  • No shared code blocks. Different complexity levels entirely.

  Layout:
  • TUI (lines 322-424): Simple LayoutContainer with math.floor/math.ceil
    row/col math, E:Scale() for sizing, centered rows, grow up/down toggle,
    single icon size per viewer
  • Ayije (Core/Layout/Shared.lua + Reanchor.lua, ~750 lines): Pixel-perfect
    system with PixelUtil, snapped metrics (ToPixelCount/PixelCountToUnits),
    separate essential row1/row2 sizes, vertical utility mode, buff group
    positioning, provisional frame readiness checks, reanchor throttling
    with retry queues, xpcall error handling
  • No shared code. Different math helpers, different layout features.

  Blizzard Frame Hooks (EXPECTED OVERLAP):
  • Both hook: viewer.RefreshLayout, viewer.itemFramePool:Acquire/Release,
    viewer.OnAcquireItemFrame, frame:SetScale(1)
  • These are the ONLY integration points for Blizzard's CooldownViewer
    system. Any addon reskinning CDM must hook these methods.
  • TUI hooks are 5-10 lines each; Ayije's are 30-50+ lines with weak-key
    tracking tables (CDM_ViewerHooked, CDM_ViewerOnAcquireHooked, etc.)
  • Hooking the same Blizzard API is not code copying.

  Container/Mover System:
  • TUI: Uses ElvUI's E:CreateMover() — standard ElvUI pattern
  • Ayije: Custom draggable containers with NineSlice overlays, EditMode
    integration, pixel-snapped position persistence, drag overlays with
    helper text, per-container lock keys
  • No overlap.

  Config UI:
  • TUI: ElvUI's AceConfigDialog, opens via E.Libs.AceConfigDialog:SelectGroup
  • Ayije: Own options addon (Ayije_CDM_Options, 22 separate Lua files)
  • No overlap.

  Visibility:
  • TUI (lines 497-522): Simple enum (ALWAYS/HIDDEN/INCOMBAT) with SetShown()
  • Ayije (Core/Fading.lua, 239 lines): Full fading system with animation
    frames, lerped alpha transitions, target-changed/ooc triggers,
    per-component fade toggles (essential/utility/buffs/bars/racials/etc.)
  • No overlap.

  Slash Command Conflict (runtime note, not code issue):
  • TUI: SLASH_TUICDM1 = '/cdm' — opens TrenchyUI config
  • Ayije: SLASH_AYIJECDM1 = "/cdm" — opens Ayije config
  • If both installed, last-loaded addon wins. Not a code-copying concern.

  UNIQUE TUI IMPLEMENTATIONS:

  • Centered Grid Layout (lines 323-415)
    Original row-centering algorithm with configurable icons-per-row,
    vertical/horizontal growth, icon zoom, and spacing.

  • Icon Zoom System (lines 116-128)
    Dual-layer icon handling (icon texture vs icon sub-texture). Custom
    TexCoord manipulation specific to CDM viewer structure.

  • Preview Text System (lines 200-264)
    ElvUI config integration showing '12' for cooldown, '3' for count in
    config preview mode.

  • DebuffBorder Suppression (lines 384-389)
    Hides and hooks DebuffBorder:Show to prevent Blizzard from re-showing.
    Not present in Ayije's codebase.

  STANDARD LIBRARY USAGE:

  LibCustomGlow-1.0 is used for glow effects — this is the standard WoW
  library for this purpose. All CDM addons use it. TUI implements 4 types
  (pixel, autocast, button, proc). Standard, not copied.

  VERDICT: No copied code. Different architecture from all compared addons.
  Ayije_CDM direct source review confirms zero shared code blocks.

  5. MOVEABLE FRAMES vs BLIZZMOVE
================================================================================

  Grade: A (Original)

  TUI APPROACH (qol.lua lines 250-358):
  • Protected frames: Uses PanelDragBarTemplate with frame-level manipulation
  • Unprotected frames: Custom drag handler with OnMouseDown/OnMouseUp hooks
  • Special handling for MailFrame and AchievementFrame with manual position
    tracking
  • Custom drag state stored in tuiDragging, tuiDragCursorX, tuiDragCursorY,
    tuiDragLeft, tuiDragTop

  BLIZZMOVE APPROACH:
  • Simple drag-and-drop with ctrl+scroll wheel scaling
  • Shift+right-click reset
  • Alt+drag detachment
  • None of these features appear in TUI

  KEY DIFFERENCES:
  • Different state management (TUI uses frame properties, BlizzMove uses
    addon-level tracking)
  • Different feature scope (TUI focuses on basic dragging, BlizzMove adds
    scaling and detachment)
  • TUI includes ElvUI Bags FreeDrag function, which is ElvUI-specific
  • TUI uses SetClampedToScreen(); BlizzMove uses different approach

  VERDICT: Original implementation. Different approach and feature set.

  6. PIXEL GLOW vs ELTRUISM
================================================================================

  Grade: A (Original)

  Both addons replace ElvUI's default aura highlight (colored health bar fill)
  with a LibCustomGlow pixel glow effect. Both use LibCustomGlow-1.0, the
  standard WoW library for this purpose. The implementations are completely
  different.

  TUI APPROACH (elvuf.lua lines 242-291):
  • Hooks UF:Configure_AuraHighlight to wrap the PostUpdate callback
  • Chains after ElvUI's own PostUpdate, then calls AfterElvUIPostUpdate
  • Hides ElvUI's fill overlay via SetVertexColor(0,0,0,0) and hiding
    AuraHightlightGlow
  • Calls LCG.PixelGlow_Start on frame.Health (the health bar specifically)
  • Pixel glow only (single type), with user-configurable lines, speed,
    thickness via DB settings
  • Uses unique glow key 'TUI_PixelGlow' for clean start/stop tracking
  • Applies to all unit frames

  ELTRUISM APPROACH (DispelGlow.lua):
  • Hooks UF:PostUpdate_AuraHighlight directly (different hook point entirely)
  • Receives (object, debuffType, _, wasFiltered) — operates on the unit
    frame object, not Health
  • Supports 4 glow types (pixel, autocast, blizzard button, proc) — mutually
    exclusive via config toggles
  • Has gradient sub-feature that iterates _PixelGlow:GetRegions() for color
    falloff
  • Excludes Player and Target frames (group/raid only)
  • Hardcodes most glow parameters (7 lines, 0.25 speed, thickness 6)
  • Tracks debuff state for gradient backdrop interaction
    (EltruismDebuffExists)

  KEY DIFFERENCES:
  • Different hook point: Configure_AuraHighlight (TUI) vs
    PostUpdate_AuraHighlight (Eltruism)
  • Different architecture: TUI wraps PostUpdate chain; Eltruism post-hooks
    a separate function
  • Different glow target: frame.Health (TUI) vs unit frame object (Eltruism)
  • Different scope: TUI is pixel-only with DB config; Eltruism has 4 types
    + gradient
  • Different parameters: TUI uses DB-driven values; Eltruism hardcodes most
  • No shared code blocks — not even similar line sequences

  VERDICT: Original implementation. No copied code. Both addons must suppress
  ElvUI's default highlight and call LibCustomGlow, but the actual code paths
  are completely different.

  7. INTERRUPT INDICATOR & FOCUS GLOW vs PLATYNATOR
================================================================================

  Grade: A (Original)

  Both addons offer interrupt-ready vs on-cooldown castbar coloring AND a
  cooldown marker on the castbar. Both use the same core WoW APIs. The
  implementations are completely different.

  TUI APPROACH (elvnp.lua lines 94-323):
  • Hooks ElvUI's NP.Castbar_CheckInterrupt as a wrapper function
  • Per-class interruptMap table, picks first known spell via
    C_SpellBook.IsSpellKnownOrInSpellBook (lines 95-122)
  • Color: ApplyInterruptColor() uses EvalColorBool() to blend between
    ready/onCD colors, also factors in notInterruptible for non-int color
  • Marker: Creates TUI_CDPositioner (StatusBar with transparent texture)
    inside a TUI_CDClipper (clipping frame). Positions marker by setting
    StatusBar value to cdDuration:GetRemainingDuration() against cast's
    total duration. Marker is a 3px ColorTexture anchored to the fill edge.
  • Reverse fill: checks castbar.channeling or castbar:GetReverseFill()
    to flip anchor direction
  • Ticker: C_Timer.NewTicker(0.1) per castbar for continuous color +
    marker updates. Ticker cancelled on cast end/interrupt.
  • Reset: ResetInterruptOverlay() cancels ticker, hides marker/clipper
  • Integrated into ElvUI's nameplate system via module hooks

  PLATYNATOR APPROACH (Display/CastBar.lua, Colors.lua, Widgets.lua):
  • Uses a mixin/widget design system — CastBarMixin with ApplyCasting()
  • Per-class interrupt spells in Display/Utilities.lua with
    GetInterruptSpellPriority() / GetInterruptSpells()
  • Color: Priority-based colorQueue system with 'interruptReady' and
    'interruptNotReady' kinds. Uses SplitEvaluate() with
    C_CurveUtil.EvaluateColorValueFromBoolean per RGBA channel for
    smooth secret-value-safe transitions.
  • Marker: Three-part widget system:
    - interruptPositioner (StatusBar tracking cast elapsed time)
    - interruptMarker (StatusBar anchored to positioner fill, tracks
      interrupt remaining duration)
    - interruptMarkerPoint (5px Texture anchored to marker fill edge)
  • Reverse fill: ReverseInterruptMarker() flips anchors for channels
  • Ticker: C_Timer.NewTicker(0.1) via frequentUpdater for color refresh.
    RefreshInterruptMarker() also runs on SPELL_UPDATE_USABLE.
  • Visibility: interruptMarker alpha set via EvaluateColorValueFromBoolean
    (hidden when ready, shown when on CD)
  • Standalone nameplate addon — not integrated with ElvUI

  KEY DIFFERENCES:
  • Different framework: ElvUI hook wrapper (TUI) vs mixin/widget design
    system (Platynator)
  • Different color system: direct SetStatusBarColor with EvalColorBool
    (TUI) vs priority colorQueue with SplitEvaluate per channel (Platynator)
  • Different marker architecture: single positioner + clipper + texture
    (TUI) vs three-part positioner + marker + markerPoint (Platynator)
  • Different marker creation: lazy GetOrCreateMarker with ColorTexture
    (TUI) vs factory-created widget with configurable asset (Platynator)
  • Different interrupt spell handling: single "best" spell (TUI) vs
    checks all known spells for "any ready" (Platynator)
  • Shared concept: both use StatusBar fill to position a tick mark on
    the castbar — this is the natural approach for this visual
  • Shared APIs: C_Spell.GetSpellCooldownDuration(), IsZero(),
    GetRemainingDuration(), GetTotalDuration(), NewTicker(0.1),
    EvaluateColorValueFromBoolean — all standard 12.x WoW APIs
  • No shared code blocks — completely different implementations

  FOCUS INDICATOR (elvnp.lua lines 325-393):
  • Creates a Frame + Texture overlay on nameplate.Health for the focus unit
  • Listens to PLAYER_FOCUS_CHANGED, NAME_PLATE_UNIT_ADDED/REMOVED
  • Uses LSM statusbar texture with configurable color (r, g, b, a)
  • Shows/hides via UnitIsUnit(unit, 'focus')
  • Always available when enabled — dedicated feature toggle

  PLATYNATOR FOCUS HIGHLIGHT (Display/FocusHighlight.lua):
  • Uses a widget/mixin system — FocusHighlightMixin with 3 methods:
    SetUnit(), Strip(), ApplyFocus()
  • ApplyFocus() calls self:SetShown(UnitIsUnit("focus", self.unit))
  • Visual is a standalone glow/border texture Frame (not health overlay)
  • Uses custom "sizedtexture" or "nineslice" assets (default: "Platy: Glow")
  • Positioned around the nameplate (not inside the health bar)
  • Part of Platynator's design system — NOT in defaults, users must add it
    manually via the Designer tab
  • Shared GetHighlight factory creates all highlight types (target, focus,
    mouseover, soft-target, etc.)

  KEY DIFFERENCES:
  • Different visual: TUI overlays the health bar; Platynator creates a
    glow/border frame around the nameplate
  • Different texture type: LSM statusbar (TUI) vs custom sizedtexture/
    nineslice glow art (Platynator)
  • Different architecture: dedicated feature (TUI) vs widget mixin in a
    design system (Platynator)
  • Different positioning: inside health bar (TUI) vs configurable anchor
    around nameplate center (Platynator)
  • Same core detection: both use UnitIsUnit("focus", unit)
  • Same events: PLAYER_FOCUS_CHANGED, NAME_PLATE_UNIT_ADDED/REMOVED
  • No shared code blocks — completely different implementations

  VERDICT: Original implementation. TUI uses a simple dedicated overlay;
  Platynator offers a more extensive widget/design system with multiple
  highlight types and configurable assets. No copied code.

  8. SKIN MODULES
================================================================================

  All five skin modules use consistent, original patterns with heavy reliance
  on ElvUI's Skins module helpers (S:HandleButton, S:HandleEditBox, etc.).

  WARPDEPLETE (skins/warpdeplete.lua — 28 lines)
  • Minimal custom code
  • Hooks WarpDeplete.RenderLayout() to apply class colors
  • Uses TUI:GetClassColor() helper
  • VERDICT: Original

  BIGWIGS (skins/bigwigs.lua — 180 lines)
  • Custom bar settings extraction from BigWigs plugins (lines 43-80)
  • LFG Timer skin with border frame creation (lines 86-151)
  • Class-colored bar backgrounds via message system (lines 155-169)
  • VERDICT: Original

  AUCTIONATOR (skins/auctionator.lua — 393 lines)
  • Comprehensive recursive skinning system (lines 152-181)
  • Heavy use of ElvUI Skins helpers
  • Custom logic for finding and skinning specific frames
  • VERDICT: Original

  BUGSACK (skins/bugsack.lua — 63 lines)
  • Straightforward frame stripping and skinning
  • Finds close button by atlas detection
  • VERDICT: Original

  OPIE (skins/opie.lua — 231 lines)
  • Most complex skin module
  • Custom descent recursion for widget skinning (lines 18-180)
  • Specialized handling for icon buttons, toggles, dropdowns, tabs
  • Scanner frame with OnUpdate for continuous skinning
  • VERDICT: Original


================================================================================
  9. OTHER QoL FEATURES vs COMMON ADDONS
================================================================================

  HIDE TALKING HEAD (qol.lua lines 14-42)
  Similar to ElvUI's internal implementation (see Section 2 note).
  Both must call the same Blizzard APIs. No alternative approach exists.
  VERDICT: Necessary pattern similarity, not copying.

  AUTO-FILL DELETE (qol.lua lines 44-68)
  Hooks StaticPopup_Show() to auto-fill DELETE_ITEM_CONFIRM_STRING.
  Common pattern used by many addons. TUI's implementation is minimal
  and straightforward.
  VERDICT: Standard pattern, original implementation.

  DIFFICULTY TEXT (qol.lua lines 71-234) vs mMediaTag / Eltruism / ToxiUI
  TUI uses a compact table-driven design with two lookup tables:
  DIFF_CATEGORY maps ~25 difficultyIDs to category strings, DIFF_LABEL maps
  categories to short text labels. Two FontStrings (label + level). ~164 lines.

  mMediaTag (mBlinkii/mMediaTag):
  • ~400+ lines, flat difficultyID-to-label map (no category abstraction)
  • Inline color escape strings vs TUI's SetTextColor()
  • Per-key-level gradient coloring via mMT:ColorFade()
  • Dungeon shortname table (~120 instanceIDs)
  • Also serves as a datatext — dual purpose
  • Checks multiple delve widget IDs (6183, 6184, 6185)
  • No shared code blocks with TUI

  Eltruism (eltreum0/eltruism):
  • ~350+ lines, massive if/elseif chain — no lookup tables at all
  • Every difficultyID handled by a separate branch
  • Gradient coloring via E:TextGradient()
  • Garrison map exclusion list, guild party detection
  • 5-second E:Delay fallback re-check
  • Hides Blizzard frames differently (iterates Minimap children checking
    .Instance, .HeroicTexture, .ChallengeModeTexture)
  • No shared code blocks with TUI

  ToxiUI (Toxicom/toxiui):
  • Does NOT have a difficulty text feature at all.

  Shared API calls (unavoidable, standard WoW API):
  • select(3, GetInstanceInfo()) for difficultyID
  • C_ChallengeMode.GetActiveKeystoneInfo() for M+ level
  • Widget ID 6183 for delve tier (Blizzard-defined constant)
  VERDICT: Original. Three different architectures (tables, flat map,
  if/elseif chain). No copied code blocks.

  FAST LOOT (qol.lua lines 236-248)
  Simple LOOT_READY event handler. Loots all items in reverse order.
  Common pattern, minimal code.
  VERDICT: Standard WoW pattern, original implementation.

  MINIMAP BUTTON BAR (qol.lua lines 360-631) vs MBB / LuckyOneUI

  MinimapButtonButton (syndenbock/MinimapButtonButton):
  • Multi-file module system with export/import pattern
  • Toggle popup model (click to show/hide) vs TUI's always-visible bar
  • Scans Minimap:GetChildren() + LibDBIcon + pattern-matching frame names
  • Mixin-based layout classes (GridLayout, HorizontalLayout, VerticalLayout)
  • Replaces 8 functions with no-ops for button locking
  • No skinning in core — separate skin modules (Default, ElvUI, Tukui)
  • Raw StartMoving()/StopMovingOrSizing() vs TUI's E:CreateMover()
  • No shared code blocks with TUI

  LuckyOneUI (Luckyone961/LuckyoneUI):
  • Does NOT have its own minimap button bar code — feature was removed.
  • Delegates to ProjectAzilroka's SquareMinimapButtons (SMB).

  ProjectAzilroka SMB (for reference, since LuckyOneUI uses it):
  • AceAddon module supporting ElvUI/Tukui/standalone
  • Scans Minimap:GetChildren() with complex exclusion heuristics
  • 6-second repeating timer to grab new buttons
  • UIFrameFadeIn/FadeOut for mouseover (vs TUI's instant alpha)
  • Replaces 8 functions with no-ops for button locking
  • Handles Blizzard buttons (mail, tracker, queue, garrison, game time)
  • No shared code blocks with TUI

  Shared commonalities (unavoidable):
  • LibDBIcon:GetButtonList() / GetMinimapButton() — the only API for this
  • Texture IDs 136430/136467 — standard WoW minimap button border/overlay
  • Alphabetical sort by GetName() — idiomatic one-line Lua
  • Grid dimension math: padding + buttons*size + gaps*spacing
  VERDICT: Original. Different architectures and feature sets.
